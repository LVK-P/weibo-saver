## 1. 项目概述

### 1.1 一句话描述

**Weibo Saver** 是一款 Windows 桌面工具，帮助用户将微博内容（文字、图片、视频）完整存档到本地，支持持续监测博文变更（编辑/删除/隐藏）并生成微博风格的可视化卡片快照。

### 1.2 核心指标

| 指标 | 数值 |
|------|------|
| 代码规模 | ~6,000 行 Python |
| 模块数量 | 25+ 模块 |
| 支持平台 | Windows 10/11 |
| 打包体积 | ~72 MB（单文件免安装 exe） |
| 技术栈 | Python 3.13 + CustomTkinter + httpx + SQLite + Pillow |

### 1.3 解决的痛点

1. **微博内容不可控**：用户发布的微博可能随时被平台删除/隐藏，无法长期保存
2. **编辑历史不可见**：微博编辑后旧版本丢失，无法追踪内容变化
3. **媒体难以归档**：图片/视频分散在 CDN，无统一本地管理
4. **批量监测困难**：手动逐条查看效率低，需要自动化轮询

### 3.2 双层执行模式

本项目采用独特的**源码/exe 双模式**架构，解决 PyInstaller 打包后的子进程调用问题：

| 模式 | 执行方式 | 隔离级别 |
|------|---------|---------|
| **源码模式** (`python gui.py`) | GUI 通过 `subprocess.run` 调用 `run_full_crawl.py` 子进程 | 完全进程隔离 |
| **exe 模式** (PyInstaller) | GUI 在后台线程通过 `runpy.run_path` 执行抓取逻辑 | Semaphore(1) 串行化 |

**设计理由**：PyInstaller 打包后 `sys.executable` 指向自身 exe，无法通过 `subprocess.run([sys.executable, "script.py"])` 调用抓取脚本。通过环境变量传递参数 + `runpy.run_path` 在进程内执行，既保持了参数传递的简洁性，又通过 `Semaphore(1)` 保证了抓取任务的串行安全。

## 5. 技术选型与决策

### 5.1 技术栈对比

| 技术 | 选择 | 备选方案 | 选择理由 |
|------|------|---------|---------|
| **GUI 框架** | CustomTkinter | PyQt6, DearPyGui | 轻量（<2MB），暗色主题内置，与 Tkinter 兼容 |
| **HTTP 客户端** | httpx (async) | requests, aiohttp | 原生 async 支持，API 风格接近 requests |
| **数据库** | SQLite + aiosqlite | PostgreSQL, TinyDB | 零配置，单文件，与 PyInstaller 打包兼容 |
| **图片处理** | Pillow | OpenCV, wand | 纯 Python，跨平台，PyInstaller 兼容性好 |
| **打包工具** | PyInstaller | Nuitka, cx_Freeze | 成熟稳定，单文件 exe 支持 |
| **文本差异** | difflib (标准库) | diff-match-patch | 零依赖，unified diff 格式标准 |

### 5.2 关键设计决策

#### 决策 1：环境变量传递参数（而非 CLI 参数或 IPC）

**问题**：GUI 和抓取引擎之间如何传递参数（Cookie、UID、配置）？

**方案对比**：
- CLI 参数：命令行可能暴露 Cookie
- IPC (socket/pipe)：增加复杂度
- **环境变量** ✅：操作系统级别隔离，进程间天然传递，PyInstaller 兼容

#### 决策 2：内容哈希去重（SHA-256）

**问题**：如何判断微博是否被编辑过？

**方案**：计算博文内容的 SHA-256 哈希值 `content_hash`，与数据库中已有值对比。不同 → 触发编辑版本保存。

**优势**：精确去重，避免因时间戳/排序变化导致的误判。

#### 决策 3：3 列正方形配图网格（220×220px）

**问题**：卡片快照中如何排列 1-9 张图片？

**方案**：3 列网格，图片居中裁切为正方形（`center_crop`），视频封面叠加播放按钮图标（矢量绘制，非贴图）。

#### 决策 4：WAL 模式 SQLite

**问题**：异步下载 + 数据库写入的并发冲突？

**方案**：SQLite WAL (Write-Ahead Logging) 模式，允许并发读取 + 一个写入，写入不阻塞读取。

### 7.2 关键查询

```sql

SELECT current_content_hash FROM posts WHERE post_id=? AND uid=?;

-- 检测删除：找出数据库中存在但本轮时间线未出现的 BID
```

## 9. 质量保障

### 9.1 线程安全策略

```
GUI 线程 (主线程)
    │
    ├── self._users       ← _users_lock 保护
    ├── self._crawling_uids ← _crawl_lock 保护
    ├── self._logs         ← _logs_lock 保护
    │
    └── ui() 入队 → _poll() 主线程分发
         ↑
         │ 所有后台线程的 UI 更新必须通过此队列
```

### 9.2 错误处理原则

| 场景 | 策略 |
|------|------|
| API 调用失败 | 记录日志 + 重试（带退避），不崩溃 |
| Cookie 过期 | `_safe_get()` 检测 403/302 → 抛出 `CookieExpiredMidCrawl` |
| 文件 I/O 失败 | 输出 `[DISK_ERR]` 日志，不静默丢弃 |
| 数据库操作失败 | 抛出 `DatabaseError` |
| 网络超时 | httpx 30s 超时 + 自动重试 |

### 9.3 反爬对抗

- 请求间隔随机化 (1.2s – 2.5s)
- 单用户每小时上限 800 次（SLA: 1000 次/小时）
- User-Agent 模拟移动端 (iPhone iOS 18)
- Referer 链完整
- XSRF-TOKEN 自动刷新

## 10. 风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 微博 API 结构变更 | 解析失败 | 中 | `raw_json` 保存原始响应，异常计数告警 |
| Cookie 频繁过期 | 无法抓取 | 高 | 扫码登录 + 自动验证 + 过期告警 |
| IP 限流/封禁 | 服务中断 | 中 | 代理池 + 限流等待 + 随机化延迟 |
| PyInstaller 打包兼容 | 打包失败 | 中 | 完整 `hiddenimports` 列表 + `_MEIPASS` 路径处理 |
| 磁盘空间不足 | 写入失败 | 低 | 写入前检查磁盘空间，`[DISK_ERR]` 告警 |
